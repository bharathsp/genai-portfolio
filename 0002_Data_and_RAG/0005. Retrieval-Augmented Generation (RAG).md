# 🔹 What is **RAG (Retrieval-Augmented Generation)?**

* **RAG** is an AI pattern that combines **Large Language Models (LLMs)** with **external knowledge sources**.
* Instead of relying only on what the model was trained on, RAG **retrieves relevant information from a database or knowledge base** and uses it to **generate accurate, context-aware answers**.

👉 Think of it as giving the AI a **real-time open book** to answer from.

# 🔹 How RAG Works

1. **User Query** ❓

   * A question is asked, e.g., *“What are the latest features in Databricks Unity Catalog?”*

2. **Retriever** 🔍

   * The system searches a **knowledge base** (like documents, PDFs, databases, or a vector store of embeddings) for relevant info.

3. **Augmentation** 📚

   * The retrieved chunks of text are **fed into the LLM as additional context**.

4. **Generator (LLM)** 🤖

   * The LLM uses both its **pre-trained knowledge** + the **retrieved context** to create a final answer.

---

# 🔹 Simple Example

**Without RAG:**
User: *“What is Databricks Unity Catalog?”*
LLM: *“Unity Catalog is for organizing data.”*
(Answer is vague, may miss details.)

**With RAG:**

1. Retriever pulls info from a company knowledge base:

   * “Unity Catalog provides centralized governance for all data assets in Databricks. Features include fine-grained access control, lineage, auditing, and data discovery.”

2. Generator (LLM) combines it with its knowledge:
   User: *“What is Databricks Unity Catalog?”*
   Answer: *“Databricks Unity Catalog is a centralized governance layer that manages data and AI assets across workspaces. It supports fine-grained access controls, data lineage, auditing, and easy data discovery.”*

👉 The answer is **richer, up-to-date, and more accurate.**

## ⚙️ Key Strategies in RAG

Let’s understand the **three main components** that make RAG work efficiently 👇

---

## 1️⃣ 🧩 **Chunking Strategy**

> Breaking large documents into **smaller, meaningful pieces** (chunks) before embedding.

### 🧱 Why Chunking?

LLMs can’t process huge documents at once (due to token limits).
So, we split documents into manageable sections to ensure each piece has **coherent context**.

### 📏 How It Works

| 📘 Original Document | ✂️ Chunked                       |
| -------------------- | -------------------------------- |
| 1000+ pages report   | Split into 500–1000 token chunks |

Each chunk is like a mini-document.

---

### 🔢 Common Chunking Strategies

| 🧩 Strategy           | 📖 Description                                                                        |
| --------------------- | ------------------------------------------------------------------------------------- |
| **Fixed Size**        | Split by number of tokens or sentences (e.g., 500 tokens).                            |
| **Semantic Chunking** | Split based on topic or paragraph meaning (using embeddings).                         |
| **Sliding Window**    | Overlapping chunks (e.g., 500 tokens with 50 overlap) to preserve context continuity. |

✅ **Best Practice:** Use overlapping chunks to avoid “cutting off meaning” between sections.

---

## 2️⃣ 🧮 **Embedding Strategy**

> Convert each chunk into a **numerical vector** (list of numbers) that represents its meaning.

### 🧠 Why Embeddings?

Computers can’t “understand” text — but they can compare vectors 🔢
So embeddings let us **find similar meanings** between user queries and chunks.

---

### 🧩 Example

| Text                                 | Embedding (simplified)  |
| ------------------------------------ | ----------------------- |
| “Capital of India”                   | [0.12, 0.76, 0.45, ...] |
| “New Delhi is the capital of India.” | [0.13, 0.75, 0.46, ...] |

They are **close in vector space** → meaning is similar!

---

### 🧮 Embedding Models

Common models:

* **OpenAI Ada-002**
* **Google Gecko**
* **HuggingFace MiniLM / BERT embeddings**
* **Instructor or E5 models (for better instruction understanding)**

---

## 3️⃣ 🔍 **Retrieval Strategy**

> Find and return the **most relevant chunks** to the user query, using the stored embeddings.

### 🗃️ Steps

1️⃣ User enters a query → converted into an embedding vector

2️⃣ Compare query embedding with all chunk embeddings (using cosine similarity 🧮)

3️⃣ Retrieve top **k most similar chunks**

4️⃣ Pass them to LLM as **context** → generate the final answer 💬

---

### 🔎 Retrieval Techniques

| 🧠 Strategy                  | ⚙️ Description                                                                 |
| ---------------------------- | ------------------------------------------------------------------------------ |
| **Vector Similarity Search** | Most common — finds nearest embeddings using cosine or dot-product similarity. |
| **Hybrid Search**            | Combines keyword (BM25) + semantic (embedding) search for better results.      |
| **Metadata Filtering**       | Add filters (e.g., by author, date, document type) to narrow results.          |
| **Re-ranking**               | Retrieve broad set first, then re-rank with another model for precision.       |

---

## 🔗 Putting It All Together

```
📚 Documents
   ↓
✂️ Chunking
   ↓
🔢 Embedding (Vectors)
   ↓
💾 Store in Vector DB (like FAISS / Pinecone / Chroma)
   ↓
🧍 User Query
   ↓
🧮 Query Embedding → Retrieve Similar Chunks
   ↓
🤖 LLM + Retrieved Context
   ↓
💬 Answer
```

## 💡 Example

**User Query:**

> “What are the side effects of Ibuprofen?”

**RAG Pipeline:**

1. Retrieve chunks from a medical PDF 📘 mentioning “Ibuprofen”
2. Embeddings find the most relevant paragraphs
3. LLM reads them and answers:

   > “Common side effects of Ibuprofen include nausea, dizziness, and stomach pain.”

✅ The model didn’t *know* this — it *retrieved* it!
